# BuildHound vs. the cdsap ecosystem — spec gap analysis

Research date: 2026-07-03.

This document compares the BuildHound specification ([build-telemetry-spec.md](../build-telemetry-spec.md)), architecture doc, roadmap, and the fourteen implemented plans (docs/plans/000–013) against the nine open-source repositories by Iñaki Villar (cdsap) profiled in [docs/research/repos/](repos/). The question it answers per area: what does prior art already solve, how well, what should BuildHound adopt, and where is BuildHound genuinely on its own.

Two framing observations up front. First, the cdsap ecosystem collectively covers almost the entire *collection* surface of the spec — task events, process health, artifact sizes, benchmark orchestration — but **no repo has any of BuildHound's product layer**: there is no versioned wire schema, no multi-tenant ingest server, no spool/retry, no auth, no regression engine, and no self-hosted dashboard anywhere in the nine repos. Second, the repos are as valuable for their scars as for their code: Talaiot's fire-and-forget publishing, build-process-watcher's security model, and Bagan's stack rot are direct evidence for decisions the spec already made, and they expose a handful of problems the spec has not yet addressed (section 4).

## 2. Per-area analysis

### 2.1 Collectors — task events

**Prior art.** Talaiot is the direct ancestor of BuildHound's TaskEventCollector and validates the core mechanism with years of production use: a shared `BuildService` implementing `OperationCompletionListener`, registered via `BuildEventsListenerRegistry.onTaskCompletion`, with everything the service needs packed into serializable `BuildServiceParameters` (`library/core/talaiot/src/main/kotlin/io/github/cdsap/talaiot/Talaiot.kt`, `TalaiotBuildService.kt`). It also demonstrates the failure modes: task outcomes parsed by string-splitting `event.displayName` instead of the typed `TaskSuccessResult` API (so no cacheable flag, no incremental flag, no execution reasons), a mutable stringly-typed `ExecutionReport` with no schema version, and reliance on internal APIs (`serviceOf`, `StartParameterInternal`, `BuildScanScopeIds`). The 2015 build-time-tracker-plugin is the same idea one generation earlier and is a case study in why per-event timestamps beat listener-side clocks: its single shared `Clock` field reset in `beforeExecute` races under `--parallel`.

**BuildHound.** Plan 005 uses the typed event API (outcome enum, incremental flag, execution reasons) — already better than Talaiot. But two Talaiot capabilities were scoped *out* of plan 005 with no follow-up plan: the task type/class dictionary and the cacheable flag.

**Adopt.** Talaiot's configuration-time `taskGraph.allTasks` dictionary (task path → class name, `_Decorated` suffix stripped), snapshotted into service parameters and deliberately degraded to an empty map under isolated projects via the public `BuildFeatures` API. Tooling events do not expose task types; this is the only public-API way to get them, and Talaiot ships the isolated-projects degradation BuildHound will need. Also adopt `util/ConfigurationPhaseObserver.kt` — see 2.1a below.

**Do differently.** Keep the typed result API, the immutable versioned payload, and zero internal APIs. Talaiot's `getModule()` labeling root tasks `no_module` and its abbreviation-expansion of requested tasks are minor conveniences worth reviewing when the tasks explorer lands.

#### 2.1a Configuration-cache state detection

Plan 003 detects CC state via per-daemon execution counters, documented as a v0 heuristic that misreports on shared daemons. Talaiot's `ConfigurationPhaseObserver` (a static `AtomicBoolean` set only by configuration-phase code, read through a `ValueSource` at build end — on a CC hit the configuration phase never runs, so the flag stays false) is a cleaner internal-API-free mechanism, proven by TestKit matrix tests (`ConfigurationCacheHit.kt`) across three Gradle versions. It should replace or corroborate the counter heuristic, and it yields configuration duration ≈ 0 on hit as a bonus signal for the still-null `configurationMs`.

### 2.2 Collectors — process probe (spec §3.6, unplanned)

This is the best-covered unimplemented area in the whole spec, with four independent proofs at different fidelity levels:

- **InfoKotlinProcess** is the exact recipe spec §3.6 needs, minus scope: wrap `jstat`/`jinfo` invocations as `ValueSource`s, hold the lazy `Provider<String>`s in `BuildService` parameters at configuration time, resolve them only at build end (`InfoKotlinProcessBuildService.kt`). Its functional tests prove CC store/reuse *and* isolated-projects store/reuse (`InfoKotlinProcessNoDVTest.kt`) — a copyable test template. All actual measurement lives in two reusable cdsap libraries, `io.github.cdsap:commandline-value-source` and `io.github.cdsap:jdk-tools-parser`, which BuildHound should evaluate for reuse or fork before reimplementing.
- **build-process-watcher** proves continuous ~5s time-series sampling (the spec's v1.x ambition) is cheap, and even ships a per-sample cost benchmark (`scripts/benchmark-jstat-metrics.sh`) — ready-made evidence for an overhead budget. Its `jstat_named_values` header-name-to-column mapping (`monitor_with_backend.sh` lines 107–141) with per-column `N/A` degradation is the right way to survive jstat output drift across JDK versions. It also documents the measurement traps (see section 4).
- **daemonitor** contributes OSHI as a cross-platform alternative to `jps`/`/proc`, PID-reuse-safe `(pid, startTime)` keying for delta-CPU, `-Xmx`/`-Xms`/GC-flag parsing (`collect/JvmArgParser.kt`), and classifier heuristics with a documented false-positive trap (`collect/GradleProcessClassifier.kt`: never match bare `~/.gradle/caches` classpaths). It empirically confirms live heap of an external JVM is not observable without attach — validating the spec's jstat/jcmd choice.
- **Talaiot** runs the same jstat/jinfo probe at publish time (`publisher/TalaiotPublisherImpl.kt`) but reads Xmx from the `org.gradle.jvmargs` *property*, not the live process — the configured-vs-used delta the spec wants needs `jinfo` per PID, as build-process-watcher does.

**Adopt.** InfoKotlinProcess's ValueSource-in-BuildService-parameters recipe and its store/reuse test template; build-process-watcher's header mapping, `ps -o rss=` (portable Linux/macOS, better than the spec's `/proc`-only mention), and `jinfo -flags` per new PID; daemonitor's classifier and arg parsing. **Do differently:** cover all three main classes (InfoKotlinProcess hardcodes `KotlinCompileDaemon` only), emit structured numbers instead of "1.2 GB" strings, and fix the GC/heap math per section 4.

### 2.3 Collectors — Kotlin reports, tests, artifacts

**Kotlin build reports and test collection have zero prior art in the nine repos.** Telltale's report menu includes Kotlin build reports and cacheable-outcome filters, but all of that is computed by external tools querying the Develocity API — nothing in the ecosystem parses KGP JSON report files or JUnit XML inside a build. These spec areas (§3.4, §3.5) proceed without a reference implementation (see section 5).

**Artifact sizes** have a complete reference: AndroidArtifactsSizeReport's `variant.artifacts.use(task).wiredWith { it.input }.toListenTo(SingleArtifact.APK/BUNDLE/AAR)` pattern with `BuiltArtifactsLoader` for split enumeration, proven CC- and isolated-projects-safe on Gradle 9.2.1 by a real E2E test. Its file-based handoff (tasks write tiny `.size` files; an end-of-build hook reads and publishes) maps directly onto BuildHound's FlowAction Finalizer. **Do differently:** emit structured `{variant, module, type, sizeBytes}` records instead of filename-encoded custom-value keys; never delete task outputs post-hoc (kills UP-TO-DATE); use `File`/`Path` APIs, not `substringAfterLast("/")` (`SizeApkTask.kt` line 39 is likely Windows-broken); and since BuildHound is a settings plugin, react to AGP per-project (e.g. `lifecycle.beforeProject`) or ship a small companion project plugin. ArtifactTransformReport additionally proves that **artifact-transform executions are unreachable from `BuildEventsListenerRegistry`** — task events only — confirming the spec's decision to park transform/cache-origin telemetry in the v1.x internal-adapters module.

### 2.4 CI environment providers

No repo solves this: Telltale and build-process-watcher are hardwired to GitHub Actions (`GITHUB_STEP_SUMMARY`, `@actions/artifact`), and Talaiot has no CI awareness at all. BuildHound's plan 002 (pure-function providers over an env map, fixed detection order, ServiceLoader extras, generic fallback, `javascript:`-scheme hardening) is ahead of everything surveyed. The one adjacent idea worth importing is daemonitor's environment fingerprinting: `AgentDetector.kt` attributes builds to AI coding agents (Claude Code, Cursor, Codex, Gemini, Aider) from env-var *names* with ambient-environment subtraction, and `SourceDetector.kt` infers terminal-vs-IDE with IDE-beats-terminal precedence. BuildHound's plugin reads the environment in-process, so tagging opt-in local builds with agent/source is nearly free and is a differentiator no incumbent (Develocity, Tuist) offers. Gradle 9.6's `--non-interactive` flag (`collect/InvocationFlags.kt`) is a corroborating signal.

### 2.5 VCS collector

Talaiot collects git branch and git user through `CommandLineWithOutputValue` ValueSources created eagerly but resolved only at publish time — the issue-#408 pattern that keeps a branch switch from invalidating the CC entry (`Talaiot.kt` lines 72–74). BuildHound's plan 004 already follows the same discipline and goes further: dirty flag, per-probe degradation, `GIT_CEILING_DIRECTORIES`/`GIT_TERMINAL_PROMPT=0`/`GIT_OPTIONAL_LOCKS=0` guards, and no plaintext git user (Talaiot ships `git config user.name` verbatim — an anti-pattern under §3.7). Nothing further to adopt here; Talaiot simply confirms the pattern works long-term in the wild.

### 2.6 Payload assembly & derived metrics

The instructive contrast is Talaiot's `ExecutionReport` (mutable, `String?`-typed durations, `java.io.Serializable`, no schema version — a poor ingest contract that the profile calls out explicitly) versus ArtifactTransformReport's analytics layer: 573 lines of pure extension functions over the raw fact list (`library/.../ArtifactTransforms.kt`) with 53 unit tests, and four presentation layers that only format. BuildHound's `PayloadAssembler` + `DerivedMetrics.compute(tasks)` in commons (plan 005) already has this shape; the repos confirm it is the right one and supply vocabulary to grow into:

- **Avoidable-miss metric**: cacheable work that executed anyway, ranked per type by duration lost (`cacheEffectivenessByTransformActionType`) — the task-level equivalent should drive BuildHound's "top cacheable misses" view once cacheable flags are collected.
- **Negative avoidance savings**: cache reuse that cost more than re-execution, aggregated with count/total/worst case — absent from the spec today and a genuinely useful cache-health signal.
- **Distribution measures per grouping** (median, nearest-rank p95, avg, max) plus `topNWithOther` bucketing, and a guarded outlier rule (>2× median, min 3 builds, zero-baseline short-circuit) as a precursor to the spec §5 median+MAD baselines.

Note that plan 005's `cacheableHitRate` currently approximates without cacheable flags ((FROM_CACHE + UP_TO_DATE) / those + EXECUTED). ArtifactTransformReport's whole taxonomy depends on distinguishing `executed_cacheable` from `executed_not_cacheable` — more evidence that the deferred cacheable-flag capture should get a plan soon, since hit-rate numbers without it quietly mix non-cacheable work into the denominator.

### 2.7 Standalone HTML artifact

ArtifactTransformReport's `cli/.../output/HtmlOutput.kt` is a working existence proof for locked decision #4 at a fidelity well beyond BuildHound's v0: a genuinely useful single-file, **zero-JavaScript** dark-theme report with headline stat cards, tables, conditional empty-hidden sections, and an inline-SVG pipeline DAG (Kahn topological ordering, longest-path levels, duration-weighted edges, `<title>` hover tooltips). The SVG-DAG technique transfers directly to the future critical-path/timeline rendering. Its superseded design doc (`docs/html-report-design.md`) also documents the inline-Chart.js alternative at ~200 KB — the fallback if interactivity is ever needed. build-process-watcher contributes the replay/compare model: a stable JSON export consumed by both the live dashboard and fully client-side replay/compare pages, with a *written* backward-compatibility policy (AGENTS.md: old exports must keep working) — a strong model for BuildHound's Comparisons page, given that the artifact already doubles as an offline payload copy. Its Plotly-from-CDN dependency is precisely what decision #4 forbids. build-time-tracker-plugin's threshold-filtered relative bar chart and "build time today" aggregate remain cheap, motivating UX ideas.

**Adopt** additionally: ArtifactTransformReport's CI smoke gate — run the real renderer end-to-end and grep the generated HTML for golden section headings (`.github/workflows/build.yml`) — a nearly free regression check for `buildhound-report` beyond the existing template tests.

### 2.8 Scrubber

daemonitor's `domain/Redactor.kt` is the only real scrubbing prior art and it corroborates plan 007's design while adding patterns BuildHound lacks: key-denylist masking applied specifically to `-P`/`-D`/`--flag=value` argument forms (plan 007's accepted limitation list explicitly includes `--token abc` flag shapes — daemonitor covers the `=` form), whole-word matching for `key` to avoid over-redacting `keystore.path`/`monkey.count`, and `scheme://user:pass@` URL-credential masking (BuildHound also has this). Two daemonitor disciplines are worth importing wholesale: the redact-before-persist invariant stated *in the storage schema itself* (`Watcher.sq` comments: values passed to inserts MUST already be redacted — BuildHound's equivalent would be a KDoc contract on `BuildStore.save`), and surfacing only exception *types*, never messages, to logs/UI because messages embed paths (`ui/live/LiveUiState.kt`) — directly relevant to the plugin's warn-and-continue failure paths and the future FailureInfo collection. Talaiot and build-time-tracker, which ship username/hostname/git-user plaintext with no opt-in machinery, are the anti-pattern that motivates §3.7.

### 2.9 Upload & spool

No repo has a spool, a retry, or an idempotency story. Talaiot's publishers swallow all exceptions into logs (`InfluxDbPublisher.publish`), and its `publishOnNewThread` spawns an executor during `BuildService.close()` — daemon shutdown — with no flush guarantee, silently dropping data. That scar is direct evidence *for* plan 008's divergence from the spec: spec §3.9 still says local uploads run on "a background thread with JVM-exit flush," which is exactly the Talaiot trap; the spec text should be amended to match the implemented synchronous-with-spool behavior (or the flush machinery must be designed explicitly, which the plan reasonably declined pre-pilot). build-process-watcher's chatty ingest (one POST per process per 5s interval, pipe-delimited strings re-parsed server-side) validates the one-gzip-JSON-document design by contrast.

### 2.10 Server persistence & auth

Greenfield with negative examples. build-process-watcher's Go backend is the anti-pattern catalog: an unauthenticated endpoint mints per-run write tokens, secrets have hardcoded dev fallbacks (`backend/internal/auth/auth.go`), a leftover unauthenticated `/test` route ships to production, and Firestore rewrites a run's whole document per ingest (1 MiB cap, racy). Talaiot's `generateBuildId=false`-by-default (InfluxDB tag-cardinality) scar is the argument for a real ingest server over raw TSDB writes; Bagan's hardcoded InfluxDB URL and rotted Grafana 6.2.5/Helm 2 stack argue for versioned schemas and env-driven config. Plan 009 (hashed tokens, tenant-from-token, bounded gzip, fail-closed auth, SQLSTATE-classified errors) already avoids every one of these. The one process idea to copy: AndroidArtifactsSizeReport's live E2E loop — build a fixture with the plugin, capture the build id via a hook, then poll the query API and assert ingested values — as the template for a full-pipeline test against a Testcontainers-hosted BuildHound server, matrixed across versions.

### 2.11 Query API & rollups

The self-hosted query API itself is unprecedented in the ecosystem, but the *read-side vocabulary* is well proven by ArtifactTransformReport (see 2.6) and by Telltale's correlation model: builds tagged with `experiment/variant/mode` at run time and re-queried by tag at report time — validation of BuildHound's `tags`/`values` design and of the `mode=benchmark, scenario, iteration` tagging in spec §7. Telltale's report menu (task-type/task-path with duration threshold, Kotlin reports, process/GC reports, cacheable-outcome filter) is close to a checklist of queries plan 010's API will eventually need to answer. Plan 010's read-time `date_trunc` rollups have no counterpart anywhere; nothing in the repos informs the materialized-aggregate decision.

### 2.12 Dashboard

No repo serves an authenticated self-hosted web dashboard. What exists: Talaiot's stale InfluxDB 1.2 + Grafana 5.4.3 demo (evidence that raw-Grafana-only is a dead end for a product), Bagan's dashboards-as-code (typed Kotlin models serialized to Grafana JSON, deployed idempotently — a pattern worth remembering if BuildHound ever offers a Grafana companion), build-process-watcher's Plotly live dashboard plus client-side replay/compare pages, and daemonitor's Compose desktop UI. Borrowable details for plan 012's successor work: forward-filled chart series with a gap cutoff of 2× the median sampling delta so dead series don't paint flat lines (`src/cleanup.ts` in build-process-watcher), and the written legacy-data compatibility policy for anything the dashboard renders from stored payloads. Bagan's automatic "winner" panel over p80/p99 comparisons is a UX precedent for the future benchmark/Comparisons views.

### 2.13 Rate limiting

Nothing in the ecosystem rate-limits anything; build-process-watcher's mintable-token endpoint is what the absence looks like. Plan 013 (nested per-token + per-source-host limiters, fail-safe env parsing) stands alone. No adoption candidates.

### 2.14 JDK toolchain & compatibility matrix

No repo does anything like plan 011's build-on-26/emit-21 arrangement. Where the repos are ahead is matrix *breadth*: Talaiot's sample workflow runs ubuntu/macos/windows with the Gradle 9 sample on JDK 17/21/25 and the Gradle 8 sample on JDK 11/15/17/21, each with `--configuration-cache` (`.github/workflows/sample.yaml`); InfoKotlinProcess's integration job spans {ubuntu, macos} × JDK {17, 19, 21} × {temurin, zulu, liberica}. BuildHound's matrix is {Gradle 8.14, 9.latest} × {CC on/off} on a single OS/vendor. Given that plan 007's only post-merge field bug was macOS-specific (temp-dir path shapes) and AndroidArtifactsSizeReport carries a likely Windows path bug precisely because no Windows job exists, a cross-OS leg — even just a cheap sample-build canary like Talaiot's — belongs in the CI matrix before OSS launch. InfoKotlinProcess also demonstrates CI dogfooding: it monitors its own builds with build-process-watcher; BuildHound should apply its own plugin to its own build as soon as the pipeline is stable enough.

## 3. Proven techniques the spec should adopt

Each item names the technique, where it's proven, and where it lands in BuildHound.

1. **`ConfigurationPhaseObserver` CC-hit detection** (Talaiot, `util/ConfigurationPhaseObserver.kt`, proven by `ConfigurationCacheHit.kt` across three Gradle versions) → replace/corroborate plan 003's daemon-counter heuristic for the `configurationCache` field; also yields a config-duration signal.
2. **Config-time task-type dictionary with isolated-projects degradation** (Talaiot, `Talaiot.kt`: `taskGraph.allTasks` → path→class map, `_Decorated` stripped, empty map under IP via public `BuildFeatures`) → the deferred task `type` field from spec §4, prerequisite for the Tasks explorer.
3. **ValueSource-wrapped external-process probes held as lazy Providers in BuildService parameters, resolved only at build end** (InfoKotlinProcess, `InfoKotlinProcessBuildService.kt`; same idiom in Talaiot) → the §3.6 ProcessProbe implementation recipe; evaluate `io.github.cdsap:commandline-value-source` and `io.github.cdsap:jdk-tools-parser` for direct reuse.
4. **The two-build store/reuse functional-test template, including the isolated-projects variant** (InfoKotlinProcess, `InfoKotlinProcessNoDVTest.kt`: assert "Configuration cache entry stored" then "…reused.") → BuildHound's compatibility contract tests for every new collector.
5. **jstat header-name-to-column mapping with per-column N/A degradation** (build-process-watcher, `monitor_with_backend.sh` lines 107–141) → JDK-version portability for the ProcessProbe; the general pattern also applies to the Kotlin build-report parser (tolerate unknown/missing fields, never index by position).
6. **`ps -o rss=` for RSS and `jinfo -flags` once per new PID for configured-vs-used heap** (build-process-watcher) → §3.6, replacing the Linux-only `/proc` mention; read the effective max from `-XX:MaxHeapSize`.
7. **`toListenTo(SingleArtifact.*)` + `BuiltArtifactsLoader` + file handoff to an end-of-build hook** (AndroidArtifactsSizeReport, proven CC+IP-safe on Gradle 9.2.1) → the §4 artifacts collector, with a FlowAction sink and structured records.
8. **Pure, exhaustively unit-tested analytics functions over the raw fact list; renderers only format** (ArtifactTransformReport, `ArtifactTransforms.kt` + 53 tests) → already BuildHound's shape; extend `DerivedMetrics` with the avoidable-miss and negative-avoidance metrics.
9. **Zero-JS single-file HTML with inline-SVG graph, stat cards, and empty-hidden sections** (ArtifactTransformReport, `HtmlOutput.kt`) → the artifact's growth path (timeline, cache panel) without ever revisiting decision #4; plus the CI grep-the-rendered-HTML smoke gate.
10. **Redact-before-persist stated as a schema-level contract; flag-argument secret forms; exception-types-only error surfacing** (daemonitor, `Redactor.kt`, `Watcher.sq`, `LiveUiState.kt`) → scrubber hardening before FailureInfo collection lands, and log hygiene for the plugin's warn paths.
11. **AI-agent and source attribution from env-var names with ambient subtraction** (daemonitor, `AgentDetector.kt`/`SourceDetector.kt`) → a cheap, additive `environment`/tags enrichment for opt-in local builds; no incumbent has it.
12. **Seed-then-measure cache isolation and tag-based experiment correlation** (Telltale: cache-write-only seed, cache-read-only fresh-runner iterations, per-mode `GRADLE_HOME_CACHE_EXCLUDES` at `runner-seed/action.yaml` lines 63–115; Develocity scan tags re-queried at report time) → the §7 benchmark-mode pipeline design; Bagan adds hermetic per-variant isolation and p80/p99 winner comparison, and the lesson to exclude failed iterations from timing series.
13. **Cross-OS × multi-JDK × multi-vendor sample-build canary and CI dogfooding** (Talaiot `sample.yaml`, InfoKotlinProcess `build.yaml`) → BuildHound's pre-launch CI matrix.
14. **Versions-to-monitor Renovate regex managers opening labeled PRs when new AGP/KGP/Gradle versions ship** (Telltale) → a cheap trigger for compatibility-matrix runs and scheduled benchmark baselines.

## 4. Gaps the repos expose in the spec

Problems cdsap's tools actually hit that the spec/plans do not yet address:

1. **Process-probe measurement math is underspecified.** Spec §3.6 says "sample `jstat -gc` … heap used/committed/max, GC time" without pinning definitions. build-process-watcher shows the traps concretely: used = EU+OU silently excludes survivors and metaspace; "capacity" is committed eden+old, not `-Xmx`; GC time as YGCT+FGCT omits CGCT and undercounts G1/ZGC. The spec should pin: read the GCT total (or include CGCT), define which memory pools count as "used," and state that capacity ≠ Xmx (the configured value comes from `jinfo`). Without this, the configured-vs-used headline metric will be quietly wrong on modern collectors.
2. **Short-lived and already-dead processes.** InfoKotlinProcess's snapshot is empty when the Kotlin daemon exits before build end or compilation runs in-process; build-process-watcher's 5s cadence misses short-lived `GradleWorkerMain` JVMs entirely. Spec §3.6's single end-of-build snapshot inherits both blind spots — they should be documented payload semantics (`processes: []` is "nothing observable," not "no processes ran"), and worker coverage expectations set accordingly.
3. **Builds that die never report.** daemonitor detects `INTERRUPTED` builds via daemon-PID disappearance — precisely the case BuildHound's in-process Flow finalizer can never see (daemon OOM-kill, `kill -9`, agent eviction). The spec's never-fail + spool design covers upload failures but not death-before-finalizer, which matters doubly because memory pressure is a headline telemetry target: the builds most interesting to a process-health product are the ones most likely to vanish. Candidate mitigations worth a spec note: a build-start marker file reconciled by the next build (spooling a synthetic "interrupted" record), and/or server-side expected-build detection via the CI connector (Timeline says a job ran Gradle; no payload arrived).
4. **Isolated Projects is tested by the ecosystem, only tracked by BuildHound.** Talaiot, InfoKotlinProcess, and AndroidArtifactsSizeReport all ship IP store/reuse tests today; Talaiot ships concrete degradation behavior (empty task-type map). BuildHound runs IP as a non-blocking CI job with no defined degradation semantics. The moment task-type capture (this document's top adoption) lands, IP is exactly what breaks it. Define the degradation contract (which fields go null under IP) and promote the IP job to blocking per collector as each one gains a degradation path.
5. **No Windows or macOS leg in CI.** Plan 007's only field bug was macOS-specific; AndroidArtifactsSizeReport's `substringAfterLast("/")` bug survives because no Windows job exists; Talaiot runs three OSes on every sample build. The scrubber, spool paths, VCS exec, and process probe are all OS-sensitive. A cross-OS sample-build canary is cheap and belongs before the pilot, not before OSS launch.
6. **Spec §3.9's background-upload text is a documented trap.** Talaiot's `publishOnNewThread` during `BuildService.close()` has no flush guarantee and drops data silently — the exact machinery spec §3.9 prescribes for local builds. Plan 008 diverged to synchronous-with-spool; the spec text should be amended so the trap doesn't get faithfully implemented later.
7. **Tag/value cardinality and size caps are missing from the plugin DSL.** Talaiot's `generateBuildId=false`-by-default exists because unbounded cardinality wrecked its backends; the research doc's risk register names custom-value cardinality abuse and spec §5 caps the metric *CLI* (100 measures, 300 chars) — but spec §3.4's `tags.put(...)`/`value(...)` DSL has no caps at all. Mirror the Datadog-style limits on the plugin side and enforce them in the scrubber/assembler.
8. **No overhead budget for the plugin itself.** build-process-watcher benchmarks its own sampler's per-call cost and publishes the script; Talaiot ships min-duration/task filters partly to bound volume. BuildHound's spec never states an overhead budget (plugin wall-time added per build, finalizer time, probe cost) or a way to measure it. Adopt a self-benchmark harness (the §7 gradle-profiler scenarios can compare with-plugin vs without-plugin) and a stated budget, since "never fail the build" is necessary but not sufficient for adoption — "never slow the build noticeably" is the real bar.
9. **`startMs` is specified but not implemented, and prior art shows it is free.** Spec §4 declares per-task `startMs`; plan 012 defers the dashboard timeline because schema v1 tasks carry only `durationMs`. Talaiot and even the 2015 build-time-tracker capture start/end from the finish event's result — the data costs nothing. Add the additive field and reconcile the spec/schema drift before more read surfaces (artifact timeline, dashboard) build around its absence.
10. **External-format drift needs concrete versioning designs, not intentions.** Three independent repos survive format drift defensively: build-process-watcher's jstat header mapping, its `ParseData` accepting 6/7/14-column records, and daemonitor's regex grammar over the explicitly-unstable daemon-log format (its top self-declared fragility). BuildHound's Kotlin build-report parser (spec §3.4/§4 `reportSchema`) has only "version the parser" as a plan; the K2 metric renames the research doc mentions already happened once. Specify the tolerance strategy (name-keyed lookup, unknown-field tolerance, per-version fixture files) before the bundler lands.
11. **Silently-skipping test suites hide coverage holes.** InfoKotlinProcess's Gradle 9.x coverage lives only in secret-gated tests that `Assume`-skip without `GE_URL`; ArtifactTransformReport's CLI is tested only via a secrets-requiring live smoke test. BuildHound already has the same shape in miniature (`disabledWithoutDocker` Testcontainers, node-absent dashboard smoke skip). Add a CI assertion that the skippable suites actually ran on at least one job, so a runner-image change can't silently zero out coverage.
12. **Docs/README drift is the ecosystem's most universal defect** — every single profiled repo has it (Talaiot's phantom Timeline publisher, InfoKotlinProcess's Gradle 7.5 claim, ArtifactTransformReport's stale Chart.js design doc, build-process-watcher's broken alternate entry points and three-way license mismatch). BuildHound already exhibits early drift (spec §1 "Gradle 8.x+" and §8's "Gradle 8.0" matrix vs the 8.14 floor; §3.9 vs plan 008; §4 `startMs` vs schema; plan cross-reference slips noted in the baseline). The existing "living document, updated in the same PR" rule covers the architecture doc but evidently not the spec — extend the rule: any plan divergence must patch the spec section it diverges from, in the same PR, the way plan 002 already did once for §3.3.

## 5. Greenfield areas — planned by the spec, proven by nobody

These are the highest-risk parts of the spec in the precise sense that no surveyed repo has done them; BuildHound's own tests are the only evidence they work.

1. **The settings-plugin + Flow API architecture itself.** None of the nine repos is a settings plugin, and none uses `FlowAction`/`FlowProviders` — Talaiot and InfoKotlinProcess both use the older AutoCloseable-BuildService-pinned-by-no-op-listener idiom, and the rest don't run inside builds at all. BuildHound's plan-000 spike (Flow API + `ServiceReference` green under CC store/reuse on 8.14) is the sole validation. The ecosystem's silence means fewer battle scars to inherit: keep the no-op-listener idiom documented as the known-good fallback, and treat Flow-API behavior across Gradle 9.x updates as a first-class regression surface in the TestKit matrix.
2. **A versioned, typed, additive wire schema with golden-file enforcement.** No cdsap repo versions its output format; Talaiot's unversioned mutable report and build-process-watcher's positional pipe-delimited lines are the state of the art being replaced. The schema-evolution discipline (never-edited golden files, `ignoreUnknownKeys`, additive-only) has no external reference — its correctness rests entirely on BuildHound's own contract tests.
3. **The entire server product layer**: multi-tenant persistence with token auth and scopes (plan 009), a self-hosted query API (plan 010), rate limiting (plan 013), and eventually retention/downsampling, admin APIs, and salt issuance. The only ecosystem "servers" are build-process-watcher's single-purpose Cloud Run backend (an anti-pattern catalog) and Bagan's provisioned InfluxDB. Everything tenancy-shaped is unprecedented and pilot risk concentrates here.
4. **Upload spool/retry/idempotency from inside a build.** No prior art at all; Talaiot proves what happens without it. Plan 008's poison-file handling, drain limits, and 4xx/5xx classification were designed from first principles — the failure-injection test suite is the only safety net.
5. **Kotlin build-report bundling** (spec §3.4) and **test-result collection** (§3.5). Zero coverage in the nine repos: nobody parses KGP JSON reports or JUnit XML in-build. The KGP-report schema-drift risk (section 4, item 10) compounds this: it is an unproven collector consuming an explicitly unstable format. The gradle.properties-validation approach (warn, don't mutate) is also untested UX.
6. **The regression engine** — rolling median+MAD baselines, budgets, verdict endpoint, alerts (spec §5). ArtifactTransformReport's guarded 2×-median outlier rule and Bagan's p80/p99 winner panel are the closest relatives, both one-shot and read-side. Continuous baseline maintenance keyed on (pipeline, task-signature, branch-class, mode), and the statistical robustness of PR-vs-baseline verdicts on noisy real CI data, are unproven anywhere in the ecosystem — this is the roadmap's Phase 2 core and its least de-risked component. The benchmark mode (§7, well-supported by Telltale/Bagan methodology) exists partly to hedge this: prioritize it accordingly.
7. **CI connector SPI + AzureDevOpsConnector Timeline pull** (spec §5). All surveyed tooling is GitHub-Actions- or Develocity-bound; nobody pulls Azure Timeline data or normalizes CI spans. The provider-agnostic SPI on the plugin side is implemented (plan 002) but the server-side connector half has no precedent and no plan.
8. **Pseudonymized identity with server-issued salts** (spec §3.7). daemonitor redacts but never pseudonymizes; Talaiot ships plaintext identity. The HMAC/truncation design is BuildHound's own, and its remaining lifecycle work (server-issued salts, rotation, `strict` mode wiring) has no external reference.
9. **Flaky detection and the quarantine closed loop** (spec §5, v1.x). The heuristics are borrowed from Bitrise/Tuist product behavior described in the research doc, but no OSS implementation exists in the surveyed set to learn from — and it is gated on the equally greenfield test collector.
10. **The metric CLI and non-Gradle-step ingestion** (spec §7, `POST /v1/metrics`). Modeled on Datadog's CLI; nothing comparable in the ecosystem. Telltale's approach to non-Gradle context (re-query Develocity by tag) is the workaround BuildHound replaces.

The overall picture: BuildHound's collection layer walks a well-trodden path and should aggressively import the ecosystem's proven mechanics (sections 2–3); its product layer — schema, server, spool, intelligence — is exactly the gap the research doc identified in the market, which cuts both ways: it is the differentiation, and it is where no one has made the mistakes yet on BuildHound's behalf.
