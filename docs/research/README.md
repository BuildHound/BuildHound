# BuildHound Research Set

Research date: 2026-07-03.

## 1. What this research set is and how it was produced

This directory contains a competitive and prior-art research pass for BuildHound, the planned self-hosted Gradle build-telemetry system (Gradle settings plugin + Ktor ingest server + dashboard). The set was produced in three stages:

1. **Repository analysis.** Nine repositories from the cdsap (Iñaki Villar) ecosystem — the most prolific open-source author in the Gradle build-telemetry space — were cloned and code-analyzed, not just skimmed from READMEs. Each analysis was adversarially verified: claims made in a first-draft summary were re-checked against the actual source, which surfaced latent bugs, dead code paths, and README drift the drafts had missed. Each repo gets its own document under `repos/` covering architecture, Gradle API usage, configuration-cache and isolated-projects behavior, test strategy, maintenance status, and concrete borrow-or-avoid guidance for BuildHound.
2. **Dashboard UX review.** 27 screenshots were analyzed — 18 Develocity build-scan screens across two scans and 9 Tuist project-dashboard screens — and merged into a single UX research document mapping both products' patterns onto BuildHound's dashboard v0 (plan 012) and the post-MVP dashboard in spec §6.
3. **Gap analysis.** Everything above was compared against BuildHound's own documentation: the spec (v0.1), the architecture doc and its decision log, the roadmap, and the 14 implemented plan files (000–013). The result is a document enumerating what the ecosystem proves, what remains greenfield, twelve spec gaps the repos expose, fourteen concrete adoption items, and ten greenfield risk areas.
4. **Feature-source research (produced in a parallel session, same date).** A deep-research pass over four reference projects outside the cdsap ecosystem — Tuist's Gradle plugin, eBay's metrics-for-develocity-plugin, Gradle's common-custom-user-data-gradle-plugin (CCUD), and cdsap's PaparazziTestDistributionExtension — via a 5-angle web sweep (24 primary sources, 118 extracted claims, 25 adversarially verified) plus three follow-up source-verification agents. Output: a CORE-vs-ADDON feature gap analysis, a tiered design for the cache-miss/build-comparison feature, and a test-sharding addon design.
5. **Cross-set reconciliation (§5 below).** The two sets were produced independently, so a final verification pass compared them and checked every disputed factual claim against BuildHound's actual source. Corrections were applied to the set-4 documents directly; errors found in the set-1–3 documents are flagged in §5 and inline in §4 (bracketed, dated) rather than silently rewritten.

## 3. Executive summary

The central finding is an asymmetry: **the cdsap ecosystem collectively proves nearly all of BuildHound's collection layer but none of its product layer.** Task-event collection via a BuildService on `BuildEventsListenerRegistry.onTaskCompletion`, configuration-cache-hit detection without internal APIs, task-type capture from the task graph, CC-safe external-process probing, and final-artifact size measurement are all battle-tested in maintained code — often with functional tests covering exactly the configuration-cache and isolated-projects scenarios BuildHound's compatibility contract demands. The riskiest technical assumptions in BuildHound's plugin are therefore already de-risked by working prior art, and in several cases (Talaiot's `ConfigurationPhaseObserver`, InfoKotlinProcess's ValueSource recipe, AndroidArtifactsSizeReport's `toListenTo` mechanics) the code is directly liftable.

Conversely, **no repo in the ecosystem has a versioned schema, a multi-tenant server, spool/retry, authentication, rate limiting, a regression engine, or a self-hostable dashboard.** Everything BuildHound built in plans 008–013 and everything planned for the Phase 2 intelligence layer is genuinely greenfield; the only "server" in the set (build-process-watcher's Cloud Run backend) is useful mainly as an anti-pattern catalog — per-sample HTTP POSTs of pipe-delimited strings, whole-document rewrites, unauthenticated token minting. This confirms the market-gap thesis from the original research: nobody open-source combines deep Gradle/Kotlin telemetry, CI context, regression intelligence, and a self-hostable backend.

The unplanned spec §3.6 process probe turns out to be the **best-covered future area**, with three complementary repos supplying the CC-safe mechanism, continuous-sampling cost evidence, jstat/jinfo portability tricks, and OSHI-based classifier heuristics — but also exposing that the spec leaves the actual heap/GC math underspecified (YGCT+FGCT undercounts concurrent collectors; capacity ≠ -Xmx).

The repos and screenshots also expose concrete gaps in BuildHound's own docs: the spec §4 `startMs` field is declared but unimplemented, blocking the dashboard timeline plan 012 deferred; spec §3.9's background-upload text describes exactly the silent-data-loss trap Talaiot ships; `cacheableHitRate` currently counts non-cacheable work in its denominator; isolated-projects is tested by three ecosystem repos but only "tracked" by BuildHound; there are no Windows/macOS CI legs; dead-daemon builds (the OOM cases a process-health product most needs) never report; and dashboard v0 has no empty states even though every pilot user starts with zero ingested builds. On the positive side, the dashboard UX research finds plan 012's direction sound: Develocity's single-build instrument and Tuist's fleet-first dashboard converge on the same patterns (two-level navigation, plain-language numbers, semantic status color, explicit zeros, honest degraded states), and the views BuildHound deferred — timeline, percentiles, comparisons, slicing — are exactly the ones spec §6 already places post-MVP.

## 4. Top 10 recommendations

Prioritized by a combination of pilot impact, cost, and how directly the source de-risks the work.

1. **Replace plan 003's configuration-cache heuristic with Talaiot's `ConfigurationPhaseObserver` trick** — a static `AtomicBoolean` read through a ValueSource, internal-API-free and proven across three Gradle versions; it yields reliable CC HIT/MISS detection and a `configurationMs` signal for free. *(Source: repos/Talaiot.md; comparison-to-spec.md.)* **[Corrected 2026-07-03 after source verification: the shipped code already implements this pattern — `DaemonState.configuredSinceLastExecution` is a static `AtomicBoolean` set only from `BuildHoundSettingsPlugin.apply`, combined with public `BuildFeatures`; the counter feeds only `daemonReused`. The "per-daemon execution counter heuristic" wording comes from plan 003's text, not the implementation. Remaining action: align plan 003's wording with the code and add the `configurationMs` capture, which is genuinely still null.]**
2. **Plan the deferred task-type and cacheable-flag capture now**, using Talaiot's configuration-time `taskGraph.allTasks` dictionary with `BuildFeatures`-based isolated-projects degradation. The current `cacheableHitRate` mixes non-cacheable work into its denominator; this is the cheapest path to honest cache metrics. *(Source: repos/Talaiot.md; comparison-to-spec.md.)*
3. **Add the additive per-task `startMs` field to schema v1.** Spec §4 declares it, the implemented schema lacks it, plan 012's timeline is blocked on it, and the UX research rates the worker-lane Gantt timeline the most legible "where did the time go" view in either incumbent product. Talaiot shows the finish event provides it for free. *(Source: comparison-to-spec.md; dashboard-ux-research.md; repos/Talaiot.md.)* **[Corrected 2026-07-03 after source verification: FALSE PREMISE — `startMs` is a required field of schema-v1 `TaskExecution` (`BuildPayload.kt:99`), populated on every task from `result.startTime` (`TaskEventCollector.kt:31`), pinned by the v1 golden file, and already used to derive `startedAt`/`finishedAt`. Plan 012's deferral note ("task entries carry durationMs but no start offsets") was wrong when committed; dashboard-ux-research.md §4.2.1 even states the opposite of its own §5. The real gaps for a worker-lane timeline are the unpopulated `worker` field (no public source in `TaskFinishEvent`; lanes can be computed greedily from start/end overlaps as a rendering concern) — the timeline is buildable from ingested data today.]**
4. **Build the §3.6 ProcessProbe on InfoKotlinProcess's recipe** — ValueSource-wrapped jstat/jinfo Providers held in BuildService parameters, resolved only at build end — evaluating cdsap's `commandline-value-source` and `jdk-tools-parser` libraries for direct reuse, and pin the measurement math the spec leaves open: read the jstat GCT total (not YGCT+FGCT), use `ps -o rss=` for portable RSS, run `jinfo` once per PID, and map jstat columns by header name. *(Source: repos/InfoKotlinProcess.md; repos/build-process-watcher.md.)*
5. **Amend spec §3.9's local background-upload text to match plan 008's synchronous-with-spool behavior.** Talaiot's fire-and-forget publish on `BuildService.close()` with swallowed errors is the documented silent-data-loss trap; the spec should stop describing it as the target design. *(Source: repos/Talaiot.md; comparison-to-spec.md.)*
6. **Fix dashboard v0's only real gap before the pilot: ship Tuist-style contextual empty states** with get-started pointers, and adopt the two cheapest incumbent readability devices — Develocity's work-avoidance ledger (count + percentage + duration triples with explicit zeros, mapping 1:1 onto the task-outcome enum) and bold natural-language count-summary sentences as section headers. *(Source: dashboard-ux-research.md.)*
7. **Define isolated-projects degradation semantics per collector and add Windows/macOS CI legs.** Talaiot, InfoKotlinProcess, and AndroidArtifactsSizeReport all test isolated projects today while BuildHound only "tracks" it non-blocking; the ecosystem also runs cross-OS × multi-JDK sample-build canaries BuildHound lacks. Promote the IP job as fields gain degradation paths. *(Source: comparison-to-spec.md; repos/Talaiot.md; repos/InfoKotlinProcess.md; repos/AndroidArtifactsSizeReport.md.)*
8. **Adopt AndroidArtifactsSizeReport's mechanics wholesale for the artifacts collector**: `onVariants` + `use().wiredWith().toListenTo(SingleArtifact.APK/BUNDLE/AAR)` + `BuiltArtifactsLoader`, proven CC- and isolated-projects-safe — swapping the Develocity sink for the FlowAction finalizer and emitting structured `{variant, module, type, sizeBytes}` records instead of filename-encoded keys. *(Source: repos/AndroidArtifactsSizeReport.md.)*
9. **Grow DerivedMetrics and the HTML artifact along ArtifactTransformReport's proven lines**: pure, exhaustively unit-tested analytics functions over the raw fact list; avoidable-miss and negative-avoidance-savings cache metrics; a zero-JS inline-SVG single-file report; and a CI smoke gate that renders a real report and greps it for golden section strings. *(Source: repos/ArtifactTransformReport.md.)*
10. **Design lost-build accounting.** daemonitor's INTERRUPTED detection (daemon-PID disappearance) covers exactly the case BuildHound's in-process Flow finalizer can never see: builds that die with the daemon. OOM-killed builds are the ones a process-health product most needs to report; a start-marker reconciliation or connector-side expected-build check closes the hole. *(Source: repos/daemonitor.md; comparison-to-spec.md.)*

Beyond the top ten, the gap analysis also recommends plugin-side tag/value cardinality and size caps (Talaiot's InfluxDB-cardinality scar; the spec caps only the metric CLI) and a measured plugin-overhead budget with a self-benchmark harness (build-process-watcher benchmarks its own sampler cost) — both cheap to specify now and expensive to retrofit later.

## 5. Cross-set reconciliation (2026-07-03)

The two research sets were produced independently in parallel sessions and then reconciled: three
comparison agents mapped overlaps/contradictions, and twelve disputed factual claims were
verified against BuildHound's actual source (file:line evidence). Verdicts:

**Confirmed against source (both sets can rely on these):**

- `cacheableHitRate` mixes non-cacheable work into its denominator (`DerivedMetricsCalculator.kt:32-36`; the KDoc admits it — plan 005). Fixed by the task-type/cacheable capture (top-10 item 2; gap-analysis item 3).
- `avoidedMs`, `criticalPathMs`, `configurationMs` are hardcoded `null` in v0 ("honest nulls", plan 005).
- Spec §3.9 still prescribes a local background-upload thread that was never built — the implementation is synchronous-with-spool everywhere (`PayloadUploader`, plan 008). Amend the spec (top-10 item 5); Tuist's `uploadInBackground` knob is then an opt-out from blocking local builds.
- `TaskExecution.type` and `cacheable` exist in schema but are never populated (scoped out of plan 005, no follow-up plan).
- Bare `CI` env var is not honored: CircleCI/GitLab/Travis/Jenkins builds are recorded `mode=local` with no `ci` block (verified in provider code + tests that pin the behavior). Fix is running as a spun-off task.
- `VcsValueSource` has no timeout on git subprocesses (`VcsValueSource.kt:51-65`; env guards mitigate common hangs but bound nothing). Fix is running as a spun-off task.
- Spec drift confirmed: §1 "Gradle 8.x+" and §8's "{Gradle 8.0, …}" matrix contradict the 8.14 floor (architecture decision log is the current truth).
- CI reality: two ubuntu-only jobs (Gradle 9.6.1 + 8.14.4 floor), Temurin only, no Windows/macOS leg, no CC on/off matrix axis (one TestKit case covers CC-off), and **the non-blocking isolated-projects job promised by the spec does not exist yet**.

**Corrections applied (each marked inline in the affected doc):**

- comparison-to-spec.md / dashboard-ux-research.md / top-10 item 3: the `startMs` "declared but unimplemented" claim is a false premise — see the bracketed correction in §4 item 3. The timeline is buildable today; only `worker` is unpopulated.
- Top-10 item 1: the shipped CC-state detection already *is* the ConfigurationPhaseObserver pattern; only plan 003's text and the missing `configurationMs` remain to fix (bracketed correction in §4 item 1).
- plugin-ecosystem-gap-analysis.md: its "Tuist upload semantics identical to BuildHound (background locally)" framing and its §7 derived-metrics list overstated the shipped state — both corrected in place.
- cache-miss-input-fingerprints.md §2.6: "CC state via BuildFeatures" sharpened to the actual mechanism (static AtomicBoolean observer + BuildFeatures).

**Cross-set synergies worth noting when planning:**

- Talaiot's `taskGraph.allTasks` dictionary (set 1) resolves the gap-analysis' open question on a CC-safe `TaskExecution.type` source; together with `cacheable` it also fixes the hit-rate denominator and feeds eBay-style by-type rollups.
- AI-agent/IDE attribution appears in both sets: daemonitor's broad prefix-matching detector (set 1) vs CCUD 2.7.0's exact env-var list, source-verified (set 4: `CLAUDECODE`, `CODEX_SANDBOX_NETWORK_DISABLED`/`CODEX_THREAD_ID`, `CURSOR_AGENT`, `OPENCODE`, `GEMINI_CLI`, `android.studio.agent`). Use CCUD's exact vars as the baseline and daemonitor's ambient-subtraction idea for robustness; note comparison-to-spec.md's "no incumbent offers this" is outdated — CCUD ships it since 2.7.x.
- Negative-avoidance-savings surfaced independently three ways (ArtifactTransformReport metric, gradle-doctor warning, android-cache-fix's disabled-tasks list) — strong signal it belongs in DerivedMetrics + server rules.
- The benchmark/experiment methodologies converge: Telltale's cache-isolation modes + tag-correlated iterations (set 1) and the build-validation-scripts five-experiment recipe (set 4) are the same design; write the §7 benchmark-mode plan from both.
- Both sets independently demand tag/value cardinality caps and confirm internal-adapters as the only route to cache origin/keys/transforms (gradle/gradle#9456 refusal + ArtifactTransformReport's unreachability finding + Tuist's internal-API usage).
