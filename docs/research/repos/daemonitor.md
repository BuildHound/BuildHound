# daemonitor (cdsap/daemonitor)

A local, privacy-first "Activity Monitor for Gradle daemons": a Compose-for-Desktop app (with a headless collector mode) that observes Gradle entirely from outside the build — polling Gradle-related JVM processes and tailing daemon logs to reconstruct individual builds and attribute them to terminals, IDEs, and AI coding agents.

Research date: 2026-07-03. Source: shallow clone of https://github.com/cdsap/daemonitor.

## Overview

Daemonitor answers two questions for a single developer machine: what Gradle activity is running right now, and what ran recently. It was explicitly built for the "agentic workflow" era in which multiple IDEs, terminals, and AI agents can drive Gradle concurrently (`requirements.md`, README). It combines two signals: OSHI-based process polling every 2 seconds (`Defaults.POLL_INTERVAL` in `src/main/kotlin/io/github/cdsap/daemonitor/Defaults.kt`) for live CPU/RSS/JVM-flag metrics, and incremental parsing of Gradle daemon `.out.log` files for authoritative per-build boundaries, outcomes, project directories, and environment-variable names. Reconstructed builds are attributed to a source (terminal vs. IDE) and, notably, to a named AI coding agent (Claude Code, Cursor, Codex, Gemini CLI, Aider) via environment-variable-name fingerprinting with ambient-environment subtraction.

Everything stays on the machine in a SQLite database. The codebase contains no HTTP client of any kind (verified by grep for `okhttp`, `ktor`, `java.net.http`, `URLConnection`, and sockets), and "no network calls" plus "no build modification" are explicit non-goals (`README.md`, `requirements.md` Non-Goals). Crucially for BuildHound: daemonitor is not a Gradle plugin and calls zero Gradle APIs at runtime — no BuildService, no Tooling API, no init scripts, no JMX. It is the inverse of an in-build telemetry plugin.

## Architecture

Daemonitor is a single-module Gradle project (Kotlin 2.0.21, Java 21 toolchain, Compose for Desktop 1.7.3 with Material 3, OSHI 6.6.5, SQLDelight 2.0.2 with the JDBC SQLite driver, kotlinx-coroutines 1.9.0 core + swing; all pinned as a mutually compatible set in `build.gradle.kts`). The code under `src/main/kotlin/io/github/cdsap/daemonitor/` is layered:

- **collect/** — `ProcessCollector.kt` (OSHI adapter, scoped to the current effective UID, skips its own PID), `ProcessSnapshotBuilder.kt` (pure snapshot construction plus delta-CPU computation), `GradleProcessClassifier.kt` (command line → `GRADLE_DAEMON` / `GRADLE_WRAPPER` / `KOTLIN_DAEMON` / `TEST_WORKER` / `JAVA_GRADLE_RELATED`), `JvmArgParser.kt` (`-Xmx`/`-Xms`/GC-algorithm flags), `InvocationFlags.kt` (`--non-interactive` / `--console=plain` automation markers), `DaemonLogWatcher.kt` (incremental offset-based log reader) and `DaemonLogParser.kt` (pure regex grammar over daemon-log lines).
- **domain/** — `BuildAggregator.kt` (per-daemon-PID state machine correlating log events with resource samples into `Build` records), `AgentDetector.kt`, `SourceDetector.kt`, `Redactor.kt`, and models (`domain/model/Build.kt`, a sealed `BuildEvent` hierarchy, `GradleProcess`, `Source`, `AgentAttribution`).
- **store/** — `WatcherDatabase.kt` (SQLDelight over SQLite; schema in `src/main/sqldelight/io/github/cdsap/daemonitor/store/db/Watcher.sq` with two tables, `process_samples` and `builds`) and `SettingsStore.kt` (a defensive properties file).
- **ui/** — Live Monitor, History, and Settings Compose screens with framework-light view models.

Two entry points share one UI-independent core. `WatcherRuntime.pollOnce()` (`WatcherRuntime.kt`) performs a full collection cycle and is driven either by `WatcherService` (desktop, `Main.kt`) or by `HeadlessLauncher` (`HeadlessMain.kt`; packaged launchers accept a `--headless` flag routed in `Main.kt`, and a `runHeadless` Gradle task exists). Data flows: OSHI poll → classify/redact → insert samples into SQLite; new daemon-log lines → redact → parse events → `BuildAggregator` windows → confirmed `Build` rows → SQLite → UI refresh. History refresh is deliberately explicit rather than reactive, because SQLDelight `asFlow` notifications proved unreliable with the JDBC SQLite driver (documented in `WatcherDatabase.kt` and `WatcherService.kt`).

## Data collected & how

Daemonitor uses no Gradle APIs. Its knowledge of Gradle comes from three external observation channels:

1. **OS process introspection via OSHI** (`oshi.SystemInfo` / `OSProcess` in `ProcessCollector.kt`): every 2 seconds it records, per Gradle-related process of the current user, the pid, parent pid, classified type, redacted command line, working directory, project path (populated for wrapper processes only, since daemon/worker cwds are runtime infrastructure — `ProcessSnapshotBuilder.kt`), CPU percent (delta of cumulative kernel+user time over wall clock, normalized by logical processor count; prior samples are keyed by `(pid, startTime)` to survive PID reuse, and the first observation yields null rather than a lifetime average), RSS in MB, parsed `-Xmx`/`-Xms`/GC algorithm, process state, and an `automated` flag from `--non-interactive` (Gradle 9.6.0+, per the `InvocationFlags.kt` KDoc) or the weaker `--console=plain`.

2. **Filesystem knowledge of Gradle internals**: `DaemonLogWatcher.kt` discovers `~/.gradle/daemon/<version>/daemon-<pid>.out.log` files and reads them incrementally with per-file byte offsets over a `java.nio` byte channel. The first read of an existing file is capped at a 256 KiB tail (the fix merged in the latest commit, issue #33), subsequent reads consume appended bytes in 16 KiB chunks, partial lines are carried across reads, CRLF is handled, and truncation/rotation resets the offset. Polling is used deliberately instead of `java.nio.file.WatchService`, whose macOS implementation lacks an FSEvents backend (KDoc).

3. **A regex grammar over daemon-log lines** (`DaemonLogParser.kt`): two line shapes — prefixed `<ISO-ts> [LEVEL] [logger] message` and the bare `BUILD SUCCESSFUL|FAILED in <dur>` outcome line — yield busy/idle daemon marks with timestamps, build-start markers (both `about to start building Build{id=..., currentDir=...}` with build id and directory, and the weaker `Starting Nth build in daemon` form with neither), environment-variable *names* (never values) from `Configuring env variables: [...]` lines, daemon identity (`uid`) and `daemonOpts` from `DefaultDaemonContext[...]`, and Gradle duration shapes (`38s`, `280ms`, `1m 2s`).

From these, `BuildAggregator.kt` reconstructs per-build records (`domain/model/Build.kt`): buildId, daemon PID plus stable daemon uid, working directory/project path, start/end times, duration, peak/average RSS and peak CPU drawn from the daemon's samples inside the busy→idle window (null for builds shorter than one poll), final status `SUCCESS` / `FAILED` / `COMPLETED_NO_OUTCOME` / `INTERRUPTED`, inferred source `TERMINAL` / `IDE` / `UNKNOWN`, a bounded redacted log excerpt (100 lines / 16,000 characters, `Defaults.kt`), and the AI-agent attribution. The per-build command line is always null by design — it does not appear in the daemon log. The ambient environment for agent-attribution subtraction is captured once from `System.getenv().keys` in `WatcherRuntime.create()`.

## Outputs & integrations

There are no publishers, backends, or report formats in the BuildHound sense — the absence is the point:

- A local SQLite database `watcher.db` (tables `process_samples` and `builds`) in the per-OS app-support directory (`Defaults.APP_SUPPORT_DIR`: `~/Library/Application Support/Daemonitor`, `$XDG_DATA_HOME/Daemonitor`, or `%LOCALAPPDATA%\Daemonitor`). On first creation the file is set to owner-only 0600 permissions and its directory is excluded from Time Machine via `tmutil addexclusion` (`WatcherDatabase.hardenFilePrivacy`); note this hardening runs only on the new-file path, not on reopen.
- `settings.properties` (retention days 1–90, default 15; appearance preference) — `store/SettingsStore.kt`. Retention purges run on startup and immediately when the setting changes.
- The desktop UI: Live Monitor (process table, headline stats, memory/`MULTI-BUILD`/`AUTOMATED` badges, per-daemon detail with a live 100-line log tail), a filterable Historical build list, and Settings.
- A headless collector mode writing to the same database.
- Native installers (`.dmg`/`.msi`/`.deb`) built by jpackage through the Compose Gradle plugin and uploaded as CI artifacts (14-day retention).
- No network output of any kind; the only build-time environment integration is `providers.environmentVariable("GITHUB_SHA")` for build-info stamping in `build.gradle.kts`.

## Techniques worth borrowing for BuildHound

Daemonitor is complementary to BuildHound rather than overlapping: it observes Gradle from outside the build with zero instrumentation, while BuildHound instruments from inside via a settings plugin and server. It uses none of the Gradle APIs BuildHound's spec centers on, so it is not an architectural model — but several concrete pieces are directly valuable.

1. **AI-agent attribution from env-var names with ambient subtraction** (`domain/AgentDetector.kt`). This is a genuinely new idea no incumbent offers. Agents are fingerprinted from environment-variable *names* only (`CLAUDECODE` / `CLAUDE_CODE*` / `CLAUDE_EFFORT` → Claude Code, then `CURSOR*`, `CODEX*`, `GEMINI*`, `AIDER*` in an ordered first-match list, with a generic `AI_AGENT`/`AI_*` marker resolving to an explicit "AI agent (unrecognized)"). Before matching, the detector subtracts the watcher's own ambient environment names, so running the tool inside a Claude Code shell does not attribute every build to Claude. BuildHound's plugin runs inside the build and can read the full environment cheaply, so tagging opt-in local builds with agent and source (Claude Code / Cursor / terminal / IDE) would be a low-cost differentiator. Only the Claude Code signature is empirically confirmed by the author; the rest are best-effort patterns. Gradle 9.6.0's `--non-interactive` flag is documented in `collect/InvocationFlags.kt` as a corroborating automation signal.

2. **Source (terminal vs. IDE) inference with IDE-beats-terminal precedence** (`domain/SourceDetector.kt`): env-name markers (VSCODE/INTELLIJ/JETBRAINS/CURSOR/etc. vs. TERM_PROGRAM/ITERM_SESSION_ID), checked IDE-first because an IDE's embedded terminal also sets `TERM_PROGRAM`; a positive marker or `UNKNOWN`, never a guess.

3. **The redact-before-persist invariant** (`domain/Redactor.kt`, `Watcher.sq`), matching BuildHound spec §3.7's scrubber requirement: every command line and log line is masked before storage or display, using a key denylist (`password`, `secret`, `token`, `credential`, `apikey`, plus whole-word `key` to avoid over-redacting names like "monkey" or "keystore") applied to `-P`/`-D`/`--flag=value` forms, and `scheme://user:pass@` URL-credential masking. The SQL schema comments encode the contract ("values passed to the insert statements below MUST already be redacted"). The surrounding hygiene — 0600 file permissions, Time Machine exclusion, current-UID process scoping, and surfacing only exception *types* (never messages, which may contain paths) to the UI (`ui/live/LiveUiState.kt` `PollError`, `WatcherService.pollSafely`) — is a good template for BuildHound's spool-directory handling.

4. **OSHI as the process probe** (relevant to spec §3.6): a mature, JNA-based, cross-platform alternative to `jps`//proc parsing for enumerating `GradleDaemon`/`KotlinCompileDaemon`/`GradleWorkerMain` JVMs, with battle-tested classifier heuristics in `GradleProcessClassifier.kt` (including a documented false-positive trap: matching a bare `~/.gradle/caches` classpath over-detects every Gradle-built app, so only genuine runtime markers count) and `-Xmx`/`-Xms`/GC extraction (`JvmArgParser.kt`) for the spec's "configured vs. used" delta. Its explicit finding that live heap occupancy is *not* externally observable without attach validates the spec's choice of `jstat`/`jcmd` for heap numbers. The PID-reuse-safe `(pid, startTime)` keying for delta-CPU sampling is worth copying.

5. **The daemon-log grammar and build-reconstruction state machine** (`DaemonLogParser.kt`, `BuildAggregator.kt`) as a plugin-less fallback reference: a busy→idle bracket is only a *candidate* window, qualified into a real build only when a positive build-start marker appears inside it (discarding phantom brackets from daemon expiration/health cycles — the plan's keystone decision KTD-1); a missing idle marker is healed by flushing on the next busy mark; a disappeared daemon PID emits an `INTERRUPTED` build. That last signal — daemon death mid-build — is precisely what BuildHound's in-process Flow-API Finalizer inherently cannot capture, so a companion doctor tool or log-based fallback could borrow this directly.

6. **Process patterns worth reusing**: docs-as-tests README screenshots rendered from synthetic state; a 3-OS CI matrix with UI tests under xvfb and packaged-binary smoke tests; and plan-traceable decision IDs in code comments. The author (cdsap) is the same as ProjectGenerator, already cited in BuildHound spec §8 for TestKit fixtures, so this repo doubles as a current sample of their engineering conventions.

## Limitations & pitfalls

- **No visibility inside a build.** Task-level, cache, test, and configuration-time telemetry are out of scope by design (`requirements.md` Non-Goals); granularity is whole-build and whole-process only.
- **The daemon log format is not a stable API.** The busy/idle/build-start/env-list/`DefaultDaemonContext` line shapes are regex-matched (`DaemonLogParser.kt`) and could break across Gradle versions; `--no-daemon` builds never appear at all.
- **Per-build command line is unavailable** (`Build.commandLine` is always null from the aggregator — "not in the daemon log"), so requested tasks and flags per build are unknown; correlating the live launcher's `AUTOMATED` flag to a specific build is explicitly deferred (`InvocationFlags.kt` KDoc).
- **The outcome line is optional enrichment**: builds whose daemon log lacks `BUILD SUCCESSFUL/FAILED` (sometimes relayed only to the client) get `COMPLETED_NO_OUTCOME`, with duration inferred from the bracket.
- **Resource attribution covers only the daemon process.** The `samplesInWindow` query joins on the daemon PID, so Kotlin-daemon and worker memory spent on a build's behalf is not attributed to it; 2-second sampling means sub-poll builds (the majority of fast incremental builds, per the plan's own measurement of ~44% sub-second in one sampled daemon) get null peaks, and CPU is null on first observation.
- **Live JVM heap is not observable externally** — only configured `-Xmx`/`-Xms` (`JvmArgParser.kt`); no jstat/jcmd/JMX sampling, RSS only. Also note `JvmArgParser` pragmatically treats a unit-less `-Xmx` value as megabytes, though the JVM's default unit is bytes.
- **Agent attribution is conservative and coarse**: it cannot distinguish two sessions of the same agent, deliberately stays silent when ambient subtraction removes all signals, and the non-Claude fingerprints are self-described as unverified best-effort patterns (`AgentDetector.kt` KDoc).
- **Redaction is best-effort**: positional/bare-argument secrets are a documented residual risk (`Redactor.kt` KDoc, plan Risks section), and a same-UID process can still read the database (SQLCipher deferred).
- **Classifier naming quirk**: every `GradleWorkerMain` JVM is labeled `TEST_WORKER`, though Gradle uses the same worker main for non-test work.
- **Local-only, ad hoc persistence**: no server, aggregation, CI story, or export; migration is attempted-`ALTER`-with-swallowed-errors (`WatcherDatabase.kt`); file-permission hardening applies only when the DB is first created; most tunables (poll interval, thresholds, log bounds) are hard-coded in `Defaults.kt` with only retention and appearance configurable.
