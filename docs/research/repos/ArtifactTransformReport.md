# ArtifactTransformReport

A CLI and Kotlin library by Iñaki Villar (cdsap) that fetches artifact-transform execution data for one or many builds from a Develocity (Gradle Enterprise) instance and turns it into console, text, CSV, and standalone HTML analytics reports.

Research date: 2026-07-03. Source: shallow clone of https://github.com/cdsap/ArtifactTransformReport.

## Overview

ArtifactTransformReport answers the question "where does artifact-transform time go in my Gradle builds?" — but it answers it *after the fact*, by querying the Develocity REST API rather than by collecting anything inside a build. Given an API key and a Develocity URL, it either queries a window of builds (filtered by project, tags, user, and so on) or a single build scan, pulls the `gradle-artifact-transform-executions` payload for each build through the author's separate `geapi-data` client library, and then computes a substantial analytics layer over the resulting in-memory list: transform durations with distribution measures (median, p95, average, max), cache effectiveness including "avoidable misses," negative avoidance savings (cases where cache reuse cost more than re-execution), plugin and module/dependency attribution, dependency-version fragmentation within a build, an inferred artifact-type pipeline graph, and per-build outlier detection. The same computed results are rendered four ways: ten picnic-formatted console tables, a concatenated text summary file, a CSV dump of every execution, and a self-contained dark-theme HTML report with an inline-SVG pipeline diagram. The library half is published to Maven Central as `io.github.cdsap:artifacttransform:0.2.0` (MIT license), and the CLI half ships as a self-executing fat binary attached to GitHub releases. For BuildHound the project is interesting not as a collection reference — it has none of the plugin-side machinery in our spec — but as a compact, well-tested example of the derived-metrics and standalone-HTML-report layers we plan to build.

## Status & maturity

The project is small but actively and carefully maintained. The last commit at research time is `63c1618`, "Merge pull request #78 from cdsap/report-improvements", dated 2026-06-26 — seven days before this research (the clone is shallow, so only that commit is visible; the GitHub PR numbering implies a longer history). The library is versioned 0.2.0 and published to Maven Central with signing configured via the vanniktech publish plugin (`library/build.gradle.kts`). The analytics layer is exhaustively unit tested: `library/src/test/java/io/github/cdsap/artifacttransform/ArtifactTransformTests.kt` contains 53 test methods covering essentially every extension function, including edge cases such as empty lists, all-zero outlier baselines, classifier-bearing GAV coordinates, and empty `changedAttributes` arrays. The `:cli` module, by contrast, has no tests at all (`cli/src` contains only `main/`); its only automated verification is a genuine end-to-end smoke test in CI (`.github/workflows/build.yml`) that builds the fat binary, runs it against a real Develocity instance for the `nowinandroid` project using repository secrets, and then greps the generated HTML for required sections. CI runs on ubuntu and macos with JDK 17 — the unit-test job on Zulu, and the fat-binary/integration job on both Zulu and Liberica. Renovate is configured with PR automerge for Maven dependencies. The ktlint Gradle plugin is applied at the root and in `:cli`, but CI only runs `test` and `:cli:fatBinary`, so lint is not actually gated.

Documentation is a mixed picture. The repository contains an unusually thorough 863-line analytics specification (`docs/new_req.md`), a design-rationale document for the HTML report (`docs/html-report-design.md`), and a structured spec-to-implementation plan with requirements traceability (`docs/plans/2026-06-24-001-feat-html-report-analytics-spec-plan.md`) — evidence of a deliberate, spec-driven workflow. However, the README and the design doc have drifted from the code: both describe an interactive report with Chart.js inlined from a bundled `cli/src/main/resources/chart.umd.min.js` resource, but no resources directory exists in the repo, and the current `HtmlOutput.kt` explicitly documents and implements a zero-JavaScript static report. The Chart.js description is stale, apparently left over from before PR #78. Minor hygiene issues linger: `settings.gradle.kts` still names the root project `TransformExecution`, the CLI module's group is `org.example`, and `cli/build.gradle.kts` declares `gson` and `kotlin-statistics` dependencies that no source file imports.

## Architecture

The codebase is a two-module Gradle build (`settings.gradle.kts`: `:library` and `:cli`, Kotlin 2.4.0, Gradle wrapper 9.6.1) with a clean separation the design doc states as a principle: "the library computes, the CLI renders."

The `:library` module contains two thin data-acquisition use cases and the entire analytics layer:

- `library/src/main/java/io/github/cdsap/artifacttransform/GetArtifactTransforms.kt` — multi-build path: one geapi-data request to discover builds matching a filter (`GetBuildsFromQueryWithAttributesRequest`), then one to fetch transform executions per build (`GetBuildsWithArtifactTransformRequest`).
- `library/src/main/java/io/github/cdsap/artifacttransform/GetSingleArtifactTransform.kt` — single-build path via `GetSingleBuildArtifactTransformRequest`.
- `library/src/main/java/io/github/cdsap/artifacttransform/ArtifactTransforms.kt` — 573 lines of pure Kotlin extension functions on `List<ArtifactTransform>` implementing every metric in the reports: grouping/duration/fingerprinting by type, outcome and avoidance outcome, distribution helpers (median, nearest-rank percentile), cache effectiveness, negative avoidance savings, provider and source-category attribution, dependency coordinates and families, fragmentation detection, per-build aggregations with outlier detection, and the changed-attribute transition graph with topological ordering and level assignment.

The `:cli` module wires clikt options into a geapi-data `Filter` (`cli/src/main/java/io/github/cdsap/artifacttransform/cli/Main.kt`) and dispatches to one of two orchestrators (`cli/.../report/ArtifactTransformReport.kt` for multi-build, `cli/.../report/SingleArtifactTransformReport.kt` for a single scan). Each orchestrator fetches the transform list once and feeds the same in-memory list to three renderers: `cli/.../view/ArtifactTransformView.kt`, which prints ten picnic console tables (Outcome, AvoidanceSavingsOutcome, TransformsByType, NegativeAvoidance, Dependency, CacheSize, Slowest, CacheEffectiveness, AttributeTransition, BuildScan views) and concatenates their rendered text into a summary file; `cli/.../output/CsvOutput.kt` for the per-execution CSV; and `cli/.../output/HtmlOutput.kt` for the standalone HTML report. There is no persistence, no server, and no state between runs — everything is one-shot processing of API responses.

## Data collected & how

The tool uses none of Gradle's in-build APIs. There is no `BuildService`, no `OperationCompletionListener` or `BuildEventsListenerRegistry`, no Flow actions, no Tooling API usage, and no process probing — it never runs inside a build. All data comes over HTTP from the Develocity REST API via the external `io.github.cdsap:geapi-data:0.3.3` client (`GEClient` + `GradleRepositoryImpl`, exposed as an `api` dependency of the library), consuming the `gradle-artifact-transform-executions` build model, which the README notes is available only in Develocity API 2023.4 and later. Coroutine-based suspend calls are bridged with `runBlocking` in `Main.kt`.

Each artifact-transform execution is one fact row carrying: `transformActionType` (the fully qualified class name of the transform action), `artifactTransformExecutionName` (whose prefix before `" ["` identifies the input source: a `project :path` module, a `group:name:version` coordinate, or a bare file name), `inputArtifactName` (the resolved artifact file), `outcome` (`success`, `failure`, `from_cache`, `up_to_date`, ...), `avoidanceOutcome` (`avoided_from_local_cache`, `avoided_up_to_date`, `executed_cacheable`, `executed_not_cacheable`, ...), `duration` and `fingerprintingDuration` in milliseconds (modeled as `String` by geapi-data), nullable `avoidanceSavings` (which can be negative) and `cacheArtifactSize`, an array of `changedAttributes` (name, from, to — the artifact-type transition the transform performs, legally empty), and `buildScanId` as the grouping dimension. The CLI filter surface covers project, tags (with `not:` negation translated to geapi's `!` syntax), user, requested task, since-build-id, include-failed-builds and exclusive-tags flags, and max-builds (default 100, hard-capped at 50,000 in `Main.kt`).

A point that matters for BuildHound: this data is not reachable from the plugin-side APIs our spec builds on. `BuildEventsListenerRegistry` delivers task-completion events only, so a `BuildService` collector never sees transform executions; transform progress events exist in the Tooling API's event model, but that requires driving the build from outside rather than a settings plugin. That is precisely why this tool leans on Develocity, and why transform telemetry, if BuildHound ever wants it, would land in the internal-adapters bucket (spec §3.1) alongside cache-origin work.

## Outputs & integrations

The tool produces four outputs from a single fetch, all written to the working directory with a timestamp suffix and a `single-` filename prefix in single-build mode:

- **Console**: ten picnic-rendered tables covering outcomes, avoidance outcomes, transforms by type (count/duration/fingerprinting), negative avoidance savings, dependency attribution, cache size, slowest transforms, cache effectiveness, attribute transitions, and per-build-scan stats (`cli/.../view/`).
- **Text summary** `summary-artifact-transforms-<ts>.txt`: the same ten tables concatenated (`cli/.../view/ArtifactTransformView.kt`).
- **CSV** `artifact-transforms-<ts>.csv`: one row per transform execution with type, duration, avoidance outcome, build scan, execution name, savings, fingerprinting, first changed-attribute from/to, and cache size (`cli/.../output/CsvOutput.kt`).
- **HTML** `artifact-transforms-<ts>.html`: a fully self-contained static report — the class doc in `HtmlOutput.kt` states "No JavaScript or external resources, so it renders offline." It opens with headline stat cards (total transforms, build scans, total duration, cache hit rate, total cache size, and avoidance savings when positive), renders the inferred pipeline as an inline SVG DAG (Kahn topological ordering, longest-path column levels, duration-weighted edge thickness, `<title>` hover tooltips on nodes and edges), and mirrors the console tables as cards, including conditional sections for negative avoidance, version fragmentation, and multi-build analysis with outlier flags. Every empty section returns an empty string and disappears from the page.

Distribution is twofold: the library goes to Maven Central for programmatic use (the README shows direct `GetArtifactTransforms(filter, repository).get()` usage), and `.github/workflows/release.yml` builds the fat binary with cdsap's `fatBinary` plugin on version tags and uploads it as a GitHub release asset, installable with a `curl` one-liner. There are no other backends or publishers — no database, no dashboard, no alerting. Incidentally, the CI workflow itself uses another cdsap tool, `build-process-watcher`, with BigQuery export, illustrating the author's broader Develocity-centric tooling ecosystem that BuildHound effectively competes with.

## Techniques worth borrowing for BuildHound

1. **"The library computes, the CLI renders."** All analytics are pure, exhaustively unit-tested extension functions on the raw fact list (`ArtifactTransforms.kt`); presentation layers only format. This is exactly the shape BuildHound's derived-metrics computation (spec §3.2 Finalizer, §4 `derived` block) should take: a pure `buildhound-commons`-adjacent module that turns a task list into metrics, testable without any Gradle or server machinery.
2. **Cache-effectiveness taxonomy.** Classifying by `avoidanceOutcome` prefix (`avoided_*` = hit, `executed_*` = ran) and singling out `executed_cacheable` as an "avoidable miss" ranked by time lost (`cacheEffectivenessByTransformActionType`) maps directly onto BuildHound's cache summary and "top cacheable misses" view. The test suite doubles as documentation of Develocity's outcome vocabulary.
3. **Negative avoidance savings.** Surfacing transforms where cache reuse cost *more* than re-execution (negative `avoidanceSavings`, aggregated per type with count, total loss, and worst single case) is a metric our spec does not have and could adopt for task-level cache analysis.
4. **Honest attribution heuristics.** Owning plugin inferred from the action's FQCN package (Kotlin/kapt/AGP/Hilt/Gradle in `provider()`), input source classified into first-party module (`project :path`), external dependency (GAV), a deliberately tiny SDK/runtime allowlist, or an explicit "Unattributed file" bucket — with a stated "do not guess attribution" rule. Reusable for BuildHound's module and task-type analytics.
5. **Distribution measures and bucketing.** Median, nearest-rank p95, average, and max per grouping distinguish frequency-driven cost from per-execution-expensive cost; `topNWithOther` keeps rankings readable. The outlier rule (total > 2× median, minimum 3 builds, zero-baseline guard in `outlierBuildScans`) is a simpler cousin of our planned median+MAD baselines (spec §5).
6. **Zero-JS standalone HTML as an existence proof.** `HtmlOutput.kt` demonstrates that a genuinely useful single-file report — dark theme, stat cards, tables, and a bespoke inline-SVG graph — needs no JavaScript at all, which trivially satisfies our locked "no CDN/network" decision (#4, spec §3.8). The superseded design in `docs/html-report-design.md` also documents the inline-Chart.js alternative and its ~200 KB tradeoff, useful when we decide how interactive our artifact needs to be.
7. **Cheap CI regression gate for the report.** The build workflow runs the real binary against a live instance and greps the generated HTML for required section headings. Grepping golden strings out of a generated report is a nearly free end-to-end check worth copying for `buildhound-report`.
8. **Pipeline inference from transition edges.** Aggregating `changedAttributes` from→to pairs into a weighted directed graph and laying it out with Kahn's algorithm plus longest-path levels (`attributeTransitionEdges`, `topologicalArtifactOrder`, `artifactLevels`, `pipelineSvg`) is a neat visualization pattern transferable to, for example, a task critical-path or module-dependency rendering.
9. **Fragmentation within one build, not across builds.** `dependencyFamiliesFragmentedWithinBuild` flags a `group:name` family only when multiple versions appear in the *same* build scan, explicitly excluding normal cross-build version drift — a subtle correctness decision worth remembering for any multi-build aggregation we do.

## Limitations & pitfalls

- **Hard Develocity dependency.** Every byte of data comes from the paid Develocity API (2023.4+). The tool has zero in-build collection capability and is useless to teams without Develocity — which is precisely the gap BuildHound targets. Nothing here helps with spec §3's plugin-side collection.
- **Cannot explain non-cacheability.** `cacheKey`, `nonCacheabilityCategory`, and `nonCacheabilityReason` exist in the endpoint but are not modeled by geapi-data 0.3.3, so the report cannot say *why* a transform missed the cache. The design doc and plan honestly record this as blocked upstream.
- **Latent CSV crash and no escaping.** `cli/.../output/CsvOutput.kt` line 23 indexes `transform.changedAttributes[0]` unguarded, although empty `changedAttributes` arrays are legal elsewhere (the library's own `attributeTransitionLabel()` handles them, and the test fixtures use `arrayOf()`); any such record throws `ArrayIndexOutOfBoundsException`. CSV fields are also written without quoting or escaping.
- **Docs/code drift.** The README and `docs/html-report-design.md` still describe a Chart.js-based interactive report with a bundled `chart.umd.min.js`; the actual `HtmlOutput.kt` is zero-JS static HTML and no such resource exists in the repo. Trust the code, not the README, when borrowing.
- **Type fragility inherited from geapi-data.** Durations and sizes are `String`s; the code mixes a safe `toMillisOrZero()` helper with roughly two dozen unguarded `.toInt()` calls (e.g. `sortedByDurationDescending`, `cacheSizeByTransformActionType`), and aggregations sum `Int` milliseconds — `Int.MAX_VALUE` is about 596 hours, and the README's own example output shows a single transform type at 1d 23h, so overflow is plausible for large build windows.
- **No tests for the CLI layer.** The only end-to-end verification requires repository secrets for a live Develocity instance, so forks and PRs from outsiders cannot exercise it.
- **One-shot, in-memory only.** No storage, no trend analysis beyond the fetched window, no server or dashboard; the multi-build view is bounded by `--max-builds` (default 100, cap 50,000) held entirely in memory.
- **Small-project hygiene.** Unused `gson` and `kotlin-statistics` dependencies in `cli/build.gradle.kts`, root project still named `TransformExecution`, CLI group `org.example`, and ktlint applied but not run in CI.
